# E2E scenarios

## Scope
Интеграционные проверки покрывают основной пользовательский путь бронирования:
гость открывает публичную страницу бронирований, выбирает день и тип встречи, выбирает слот,
заполняет форму контактов и получает подтверждение.

Документ описывает сценарии для браузерных e2e-тестов, где frontend и backend работают вместе.
Источник истины по поведению API: `spec/`.

## Scenario 1 — Successful booking
Предусловия:
- в системе есть хотя бы один доступный для бронирования `event type`;
- в ближайшие 14 дней для него есть хотя бы один свободный слот;
- публичная страница бронирований доступна.

Шаги:
1. Открыть публичную страницу бронирований.
2. Выбрать день, для которого доступна запись.
3. Нажать `Записаться`.
4. Если открывается шаг выбора типа встречи, выбрать `event type`.
5. Выбрать доступную дату и слот.
6. Заполнить форму гостя: `name`, `email`.
7. Подтвердить бронирование.

Ожидания:
- создаётся `booking`;
- пользователь видит экран подтверждения;
- на экране отображаются название встречи, дата/время и `email` гостя.
- при повторной загрузке публичной страницы этот слот больше не доступен для новой записи.

## Scenario 2 — Slot becomes unavailable before submit
Предусловия:
- существует `event type` с одним доступным слотом;
- гость дошёл до шага ввода контактов для этого слота.

Шаги:
1. Открыть публичную страницу бронирований и выбрать слот.
2. До отправки формы сделать слот занятым другим действием:
   через API, фикстуру или подготовительный шаг backend-состояния.
3. Вернуться в браузер.
4. Отправить форму бронирования.

Ожидания:
- `booking` не создаётся;
- пользователь остаётся в потоке бронирования;
- отображается ошибка конфликта, а экран успеха не показывается.

## Scenario 3 — Validation errors
Предусловия:
- открыт шаг ввода контактов.

Шаги:
1. Оставить обязательные поля пустыми или ввести некорректный `email`.
2. Нажать submit.

Ожидания:
- запрос на создание `booking` не уходит;
- `booking` не создаётся;
- отображаются ошибки валидации.

## Notes For Playwright
- Основной обязательный сценарий шага: `Scenario 1`.
- `Scenario 2` и `Scenario 3` полезны как минимальный набор негативных интеграционных проверок.
- Для стабильных e2e-тестов нужен контролируемый backend-state:
  фиксированные `event types`, расписание и свободные слоты.
- В CI тесты должны поднимать frontend и backend вместе и запускаться после unit/integration проверок.
